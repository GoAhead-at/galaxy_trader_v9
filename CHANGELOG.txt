GalaxyTrader - Changelog
========================================


Version 0.13.3 (2026-04-17)
FIX: MK3 + MK4 `matching_work_loop` (live-pipeline scoring) now applies the same priority-ware multiplier that `cache_work_loop` already applied (DOCS/MK3_PIPELINE_REFACTOR.md §3.10). Previously, fleets with `$prioritywares` configured got priority ordering on cache hits but distance-optimized / priority-blind ordering whenever the live/matching pipeline produced the trade, which made the user-visible "priority ware first" guarantee depend on whether the cache had a fresh entry for that workunit. Resolution now happens once per workunit via `lib.gt.priority_wares.resolve` (MK3 passes `['GalaxyTraderMK3']`, MK4 passes `['GalaxyTraderMK3', 'GalaxyTraderMK4Supply']`) and the resolved priority list + clamped multiplier (125-300) are applied per pair to `baseScore`, with a `PriorityBoosts` count emitted in the post-loop debug summary.
PERF: MK3 + MK4 `prefilter_work_loop` route-legality probing (`run_script lib.gt.pathfinding mode=route_legality`) is now memoized per chunk on the `(buySector, sellSector, ware)` triple (DOCS/MK3_PIPELINE_REFACTOR.md §3.9). `lib.gt.pathfinding` already maintains a persistent global cache keyed on `(home, buy, sell, ware)` with `$BlacklistEpoch` invalidation, so the per-chunk memo only saves the `run_script` interpreter overhead (table construction + child-script context switch) when many rows in the same chunk share the same triple — common when a station appears in multiple offers for the same ware. Memo lives on locals (auto-freed at the cooperative `<wait>` yield); a running total of skipped child-script calls is held on `this.$pfRouteMemoHits` so the post-chunk summary debug can report it (`routeMemoHits=N`), then `remove_value`d when the prefilter completes.
REFACTOR: MK3 + MK4 init / main_loop attention-lifecycle duplications lifted into shared library helpers (DOCS/MK3_PIPELINE_REFACTOR.md §3.11). Four new `lib.gt.attention.*` scripts were added:
  - `lib.gt.attention.min_player_money` — replaces ~10 inline `do_if value="not this.$minPlayerMoney?"` resolve blocks (each duplicating the same `?-and-!=null-and-gt 0` guard against `global.$GT_GlobalSettings.$Fleet.$MinPlayerMoney` with a 1000-credit fallback) at every money gate, fast-money short-circuit, and budget-clamp site in MK3 + MK4.
  - `lib.gt.attention.is_gt_commander` — collapses the (`order == X` OR (`order == 'Assist'` AND `baseorder == X`)) idiom that was copy-pasted ~9 times across MK3 (init / main_loop / Phase 6 commander-delta branches / subordinate orphan check / Phase 7 name-restoration) and again in MK4 with `'GalaxyTraderMK4Supply'` added; future GT order variants now extend `validOrderIds` in one place.
  - `lib.gt.attention.restore_original_name` — encapsulates the GT_Pilots-lookup + `OriginalShipName` extraction + `signal_objects param='GT_Ship_Order_Aborted'` payload (with optional `skipIfStillGT` short-circuit that respects both direct GT order AND a GT-commander chain via `lib.gt.attention.is_gt_commander`). Replaces the Phase 7 / commander-delta name-restoration blocks in MK3 (init Phase 7, main_loop Phase 7, main_loop commander-delta) and MK4 (same three sites). The orphan / missing-pilot / subordinate-cleanup sites that fall back to `@this.ship.macro.name` are intentionally left inline — those are a different scenario and not in the lifecycle dedup scope.
  - `lib.gt.attention.emit_gt_state_changed` — fires the matched pair of `GT_Ship_State_Update` + `GT_Commander_GT_State_Changed` signals that every GT assignment / cancellation site emits, with a free-form `source` tag so MD-side traces can attribute each emission to the AI site that produced it. Replaces 4 inline emit pairs in MK3 init / main_loop (`AI_INIT_ASSIGN`, `AI_INIT_REASSIGN`, `AI_LOOP_REASSIGN`, `AI_LOOP_CANCEL`) and 4 more in MK4. The payloads were byte-identical aside from the `source` string, making this the highest-payoff dedup of the attention-lifecycle group.
FIX: MK3 + MK4 `start_trading` registration wait is now bounded (was an unbounded `do_while value="not global.$GT_Ships..."` with `wait exact="500ms"`). Spins for max 20*500ms = 10s, then re-emits `GT_Register_Ship` once and waits another 10*500ms = 5s. If still unregistered after that, logs a warning and `<resume label="main_loop"/>` so the next iteration can retry instead of pinning the AI block forever on a queue stall / lost signal. Counter is held on `this.$regSpinCount` because locals do not persist across `<wait>` yields.
FIX: MK3 + MK4 `escape_from_blacklisted` MoveWait `timeout` was hard-coded to `1s`, which caused the order to settle within a single AI tick after arrival; the next `main_loop` pass then re-blacklisted and re-fired MoveWait in a tight loop. Timeout is now `if $jumppath? and $jumppath.count gt 0 then $jumppath.count * 60s else 60s`, giving the ship a realistic settling window before GT re-evaluates. Misleading "small timeout so GT resumes shortly after arrival" comment replaced with the actual semantics (the AI resumes immediately because of `immediate="true"` + `<resume>`; `timeout` is the post-arrival hold).
REFACTOR: Extracted shared `lib.gt.priority_wares.resolve` library script that walks `instanceWares` -> `localWares` -> `commander.defaultorder.$prioritywares` (or `commander.defaultorder.$baseorder.$prioritywares` via Assist) once and returns `[]` when no source is non-empty. Replaces the ~1.5KB nested ternary previously inlined at `order.trade.galaxytrader.xml:3843` and `order.trade.galaxytradermk4.xml:4190` (each evaluated the same `?-and-!=null-and-.count gt 0` triple-guard 5x and re-walked `commander.defaultorder` 4x per resolve). The `validOrderIds` parameter accepts `['GalaxyTraderMK3']` for MK3 and `['GalaxyTraderMK3', 'GalaxyTraderMK4Supply']` for MK4. Also unblocks Tier C item #9 (matching-loop priority parity).
REFACTOR: Removed unreachable `start_repair` label and `global.$GT_AutoRepair`-driven dispatch in MK3 (`order.trade.galaxytrader.xml`) and MK4 (`order.trade.galaxytradermk4.xml`); the corresponding `do_elseif` arm in `md/gt_maintenance.xml` (auto-repair path) is also removed. The global was never written by any script, so the label was dead code that only inflated the script footprint.
REFACTOR: Removed orphan `signal_objects param='GT_UICreditsDue_TradeOrder'` emits from MK3 and MK4 trade-completion logging; no MD/UI cue listened on this param, so the per-trade table allocation and `signal_objects` call were pure overhead.
REFACTOR: Removed unused `$maxDistance = $workUnit.$MaxDistance` extraction in `process_live_offers_workunit` and `process_trade_matching_workunit` (MK3 + MK4); the local was never read in either workunit body (sector pre-filtering happens upstream in cache mode and matching uses a 0/1-jump heuristic), so the lookup was wasted work.
PERF: `[GT-LogbookDump]` and the `Signal received during market update wait` `chance="100"` debug_text in MK3 + MK4 are now wrapped in `do_if @global.$GT_GlobalSettings.$Debug.$EnableDebugLogging`, eliminating the highest-volume unconditional log spam (one full multi-line logbook string per ship per retry, plus one line per market-update interrupt).
FIX: MK3 + MK4 `cache_work_loop` now resets `$cacheIsPriority` to `false` per cache-entry iteration and unconditionally assigns `$bestWasPriority = $cacheIsPriority` when a new best is selected; previously the local could carry over `true` from a prior priority match, falsely flagging non-priority winners as priority and skewing the post-search summary log.
FIX: GT Diagnose `$minPlayerMoney` source aligned with the new `lib.gt.attention.min_player_money` AI helper. The diagnostic was reading `global.$GT_AIParameters.{$ship}.$MinPlayerMoney`, but `$GT_AIParameters` is only populated by `GT_RequestRegistry_Acquire` with the trade-search parameter table, which never includes `$MinPlayerMoney` (only `$IgnoreBuildStorage` / `$IgnoreCarrierAux` / `$HomeBase` / `$TraceId` / `$DistancePenalty` / `$AutoDeployResourceProbes` / `$MaxDistance`). As a result the gate threshold defaulted to 0 + 100000 Cr regardless of the user's actual fleet setting, producing a misleading "Money gate — player funds too low" verdict whenever the player was below 100000 Cr but actually above their configured threshold (and missing it when the threshold was set higher). Read order now mirrors the AI helper: `global.$GT_GlobalSettings.$Fleet.$MinPlayerMoney` (with the same null/`gt 0` guard and 1000-credit fallback), with the legacy `$GT_AIParameters.{$ship}.$MinPlayerMoney` kept as an upper-bound override so a future per-ship writer would still take effect.
FEATURE: Added "Miner Resource Scan" section to GT Diagnose for ships running `GalaxyMiner` (direct or via Assist commander). The diagnostic tool runs its own dedicated scan independent of the miner AI snapshot, so the section produces meaningful data immediately on open regardless of where the miner currently is in its loop (`MiningCollect` / `Sell` / `FindResource` / wait phases) or whether it has ever emitted a `GT_Miner_Diag_Update`. Answers "what mineable resources are actually within gather range and which sectors host them" — the most common question when miners idle on "No mineable wares".
  - MD side (`gt_context_diagnose.xml` SECTION 3b): for `$orderId == 'GalaxyMiner'`, the diagnostic (1) computes effective gather range from `$gtOrder.$maxGatherDistance` (or 0 when `$stayAtHome`), (2) picks reference sector — `$gtOrder.$homeSector` if set, else current ship sector (super-highway-aware via `@$ship.zone.destination.sector`), (3) enumerates sectors via `find_sector space="player.galaxy"` + `match_gate_distance object="$rscRefSector" min="0" max="$rscEffectiveRange"`, (4) applies the same A4 safety pre-filter the miner uses (hostile-owner via `$ship.owner.hasrelation.enemy.{$sector.owner}`, `hashazardousregion` + `containsthewave` + `takesregiondamage`, `iscontested` + `contestingfactions.mayattack.{$ship}`, sectoractivity `hasblacklist` + `isblacklisted`), always keeping the current ship sector and commander sector, (5) builds the candidate ware list with a 3-level fallback chain that mirrors `order.trade.galaxyminer.xml:493-559` — primary `ship.warebasket.list`, secondary `ship.cargo.list` (for ships without an explicit basket that already carry minable wares), tertiary `$diagMinerOfferWares` (the diagnose miner-offer pool already computed by section 3 from the miner's own derivation chain: `$diagMinerSelectedWare` + primary-station buy offers + cargo) — all filtered to `hastag.minable` + `solid/liquid` waretransport + `cargo.max gt 0`, and (6) probes each (kept-sector, ware) pair with `sector.yieldrating.{ware} ge 1` + `sector.resources.{ware}.exists` (the exact same predicates the miner uses at order.trade.galaxyminer.xml ~717-722). Header rows surface reference sector, effective gather range, sectors discovered, sectors kept after A4 filter, sector blacklist group, kept sector names (capped at 8 with "+N more" overflow indicator), min yield rating, candidate ware count + source label (`ship.warebasket` / `ship.cargo.list` / `diagnose miner-offer pool ...`), and reachable ware count. A 6-column structured row block then lists wares sorted by best yield desc (reachable first, unreachable grouped at the end) with `Status | Ware (* = active) | Best Yield | Reachable | Sectors | Notes`, top 12. Sorting uses an in-place selection sort over a `$rscClaimed` boolean parallel list (X4 MD has no `insert_in_list`). The `* active` marker is the only field still pulled from the miner snapshot (`$minerDiag.$SelectedWare`), as a pure overlay. NOTE: X4 expression rule — `@$x.y?` is invalid (cannot combine the `@` error-suppression operator with the `?` existence check); existence-then-read is split into `if $x.y? then @$x.y else fallback`, and nested chains like `$ship.zone` are guarded with a single `@$ship.zone.<final>` (the engine returns null on the whole chain if any step fails) instead of `@$ship.zone? and @$ship.zone.<next>?`.
  - AI side (`order.trade.galaxyminer.xml`): no scoring changes. The `signal_objects param='GT_Miner_Diag_Update'` payload remains the lightweight runtime-state set (selected ware, primary station, urgency, overflow flags, supplier / intent / target counts, intent phase, timestamp); the diagnostic does not need any extra fields because it runs its own scan.
  - Lua side (`ui/gt_context_diagnose.lua`): added `isMinerScan` layout case so the new tab gets the proper column headers (`Status | Ware (* = active) | Best Yield | Reachable | Sectors | Notes`), column widths (6/20/10/10/10/flex), right-aligned Best Yield + Sectors, and centered Reachable, instead of falling through to the generic "Col 1, Col 2, ..." render.
HARDENING: `gt_context_diagnose.xml` report serializer (`$report.$Sections` -> delimited string for Lua transfer) and debug-log dumper now skip null rows defensively (`do_if value="$row?"` / `do_if value="$logRow?"`). Previously, if any section builder produced a null entry in its `$Rows` list — most commonly when an XML expression parse-time error made an `append_to_list exact="table[...]"` evaluate to null — the downstream `$row.$Status` / `$logRow.$Status` property lookup raised three errors per consumer per null row and printed `[null] null: null` lines to the in-game log, masking the real diagnostic output. With the guard, the broken section degrades to "missing one row" instead of taking down both consumers and the entire log dump.
FIX: GT Diagnose "Miner Resource Scan" gather range now applies the same pilot-skill clamp the miner AI uses at runtime, instead of trusting the raw `$gtOrder.$maxGatherDistance` order param. The miner clamps the param against `$pilotMaxParamDistance` at init (`order.trade.galaxyminer.xml` lines 191-214) and again against `$pilotMaxJumps` per execution slice (lines 1218-1245), using `global.$GT_Pilots[pilot].$Level` with the mapping {<=2:1, <=5:3, <=8:5, <=11:10, <=13:15, else:25}. Diagnostic now mirrors that mapping verbatim, so a low-skill miner with `maxGatherDistance=5` correctly reports "Gather Range: 1 jumps (your setting: 5, limited by pilot skill to 1)" and only enumerates the sectors actually within the effective 1-jump radius — the previous implementation showed up to 5-jump-distant sectors that the miner could never reach. Stay-At-Home still wins over the pilot cap (forces 0). Pilot source is `global.$GT_Pilots` (same source the miner uses), not `$ship.pilot.skill.management` (used by the trader section above), because the miner does not consult the management skill directly.
FIX: GT Diagnose "Miner Resource Scan" reference-sector resolution now matches the miner AI's own anchor sources. Previously the diagnostic only checked `$gtOrder.$homeSector` (which is the Assist-resolved order, sometimes the commander's order rather than the ship's own), so subordinate miners and station-assigned miners were rendered as `Reference Sector: <current sector> (current ship location, no home base set)` even though `order.trade.galaxyminer.xml` init (lines ~150-162) auto-anchors every miner at registration via `edit_order_param`. Resolution chain now walks all six anchor sources in priority order: (1) `$gtOrder.$homeStation.sector`, (2) `$gtOrder.$homeSector`, (3) `$ship.defaultorder.$homeStation.sector`, (4) `$ship.defaultorder.$homeSector`, (5) `$ship.order.$homeStation.sector` / `$ship.order.$homeSector`, (6) `$ship.commander.sector` if the commander is a station — only falling through to current ship sector when no anchor of any kind exists. The label now reports which source resolved (`(home base: <station name>)` for station anchors, `(home sector)` for sector anchors, `(current ship sector)` only as the last-resort fallback) instead of the misleading "no home base set".
POLISH: GT Diagnose "Miner Resource Scan" rows are now player-facing only — internal variable names, MD action names, and filter codes that previously leaked into `$Detail` strings have been replaced with plain language. `(gtOrder.$maxGatherDistance)` removed from Gather Range; `via find_sector + match_gate_distance` removed from Sectors In Range; `Sectors Kept (A4 filter)` -> `Safe Sectors` with "(hostile, hazardous, contested, or on your sector blacklist)" instead of the `A4 filter` tag; `Min Yield Rating` -> `Minimum Yield Rating` with "(sectors with a lower rating are ignored)" instead of "mirrors miner this.$minReachableYieldRating; sectors below this are pruned"; ware-source labels reworded to "ship ware basket" / "current ship cargo" / "minable wares your stations buy" instead of `ship.warebasket` / `ship.cargo.list` / `diagnose miner-offer pool (selected ware + primary-station buy offers)`; `Reachable Wares` detail rewritten to "X of Y wares can be mined in a safe sector in range"; `--- Self-Scan Context (independent of miner AI snapshot) ---` -> `--- Scan Context ---`; `--- Wares (sorted by best yield desc, top 12) ---` -> `--- Wares (best yield first, top 12) ---`; the no-data fallback now reads "No minable wares are available to this miner. Add a minable ware to the ship ware basket, give the ship a solid or liquid cargo bay, or assign a station that buys a minable ware (see Sell Wares above)." instead of referencing `ship.warebasket` / `miner-offer pool` / `Diagnose Sell Wares`. Stay-At-Home is now spelled out instead of `(stayAtHome -> 0)`. Reference Sector home/current labels reworded to `(home base)` / `(current ship location, no home base set)`. Per-ware "Notes" column changed from "no kept sector with yield >= N" to "no safe sector in range has yield >= N".
